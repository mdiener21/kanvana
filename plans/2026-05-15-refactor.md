ARCHITECTURAL FRICTION ANALYSIS
================================

FILE SIZES & EXPORTS
====================

Core State & I/O:
- storage.js: 1161 lines, 39 exported functions (huge, monolithic state hub)
- tasks.js: 513 lines, unknown export count
- task-modal.js: 790 lines, 8 exports (+ 6 state getter/setters)

Rendering Layer:
- render.js: 223 lines, 4 exports (delegator)
- swimlane-renderer.js: 283 lines (specific renderer)
- column-element.js: 345 lines (specific renderer)
- task-card.js: 335 lines (specific renderer)

Infrastructure:
- dragdrop.js: 348 lines, 1 main export (initDragDrop)
- swimlanes.js: 723 lines (complex swimlane logic)
- modals.js: 118 lines (facade/delegator)
- columns.js: 133 lines, simple CRUD
- activity-log.js: 21 lines (pure factory)
- events.js: 35 lines (event bus)
- normalize.js: 127 lines (validator lib)

FRICTION POINT #1: storage.js is a GOD MODULE
==============================================
- 1161 lines with 39 exported functions
- Acts as both:
  * Async IDB abstraction (getDB, schedulePersist, etc.)
  * In-memory state container (loads/saves for all entities)
  * Sync/migration logic (normalizeBoardModelIds)
  * Deleted-item tracking (loadDeletedTasksForBoard, purgeDeleted)
- Every module that needs data imports from storage (16 modules)
- Callers must understand state structure:
  * Board context (getActiveBoardId, setActiveBoardId)
  * Global vs per-board loading (loadTasks vs loadTasksForBoard)
  * Lazy-load patterns for per-board data
- No single API or facade — each caller directly calls load* / save*
- Hard to test: requires IDB setup even for pure state logic

FRICTION POINT #2: task-modal.js is a DOM-STATE HYBRID
========================================================
- 790 lines mixing:
  * DOM manipulation (modal open/close, field binding)
  * State management (selectedTaskLabels, returnToTaskModalAfterLabelsManager)
  * Cross-modal coordination (getSelectedTaskLabels, setSelectedTaskLabels)
- State is exposed via 6 getter/setter exports — leaks implementation
- modals.js coordinates 4 modal sub-modules by importing their getter/setters
  (example: line 67-76 in modals.js)
- Hard to refactor: state lives in task-modal but is "owned" by modals.js facade
- No separation of concerns: task-modal both renders AND manages state

FRICTION POINT #3: render.js is a THIN DELEGATOR
==================================================
- 223 lines, only 4 exports (setBoardFilterQuery, syncTaskCounters, syncCollapsedTitles, renderBoard)
- But renderBoard() orchestrates 7 other modules:
  * Loads from storage (loadColumns, loadTasks, loadLabels, loadSettings)
  * Delegates to swimlane-renderer.js or renderStandardBoard()
  * Then calls initDragDrop, updateColumnSelect, renderIcons, refreshNotifications
- Callers DON'T import render directly to avoid circular deps; instead use events.on(DATA_CHANGED)
- This "event bus workaround" is a sign render.js is entangled

FRICTION POINT #4: swimlanes.js is UNDEREXPOSED
=================================================
- 723 lines but few public boundaries
- Mixed concerns:
  * Swimlane config (SWIMLANE_GROUP_BY_LABEL, SWIMLANE_GROUP_BY_PRIORITY)
  * Swimlane state (collapsed lanes, group-by selection)
  * Swimlane layout logic (groupTasksBySwimLane, buildBoardGrid)
  * Swimlane UI controls (syncSwimLaneControls, initializeSwimLaneControls)
- Callers must know internal structure:
  * swimlane-renderer.js imports getVisibleTasksForLane, isSwimLaneCollapsed, toggleSwimLaneCollapsed
  * Render.js calls syncSwimLaneControls(settings) — assumes swimlanes understands settings shape

FRICTION POINT #5: PASS-THROUGH COORDINATORS
==============================================
- modals.js (118 lines): pure re-export + facade
  * Does NOT implement modal logic; just routes to sub-modules
  * All public functions are re-exports or thin wrappers
  * setupModalCloseHandlers() is utility but trivial
  * Main value: Escape key handler and cross-modal state wiring
- render.js (223 lines): thin orchestrator
  * Creates columns, tasks, swimlanes; initiates dragdrop
  * Doesn't implement business logic; just loads and delegates
  * Main complexity: filter logic (taskMatchesFilter) is in the module

FRICTION POINT #6: STATE ACCESS PATTERN IS CHATTY
===================================================
- render.js loads from storage 4 times per renderBoard():
  const columns = loadColumns();     // line 183
  const tasks = loadTasks();         // line 184
  const labels = loadLabels();       // line 185
  const settings = loadSettings();   // line 186
- Each load unpacks state and returns a new object
- Callers never know if they have stale state (no lifecycle hooks)
- No batching: each module does its own load* calls
  * dragdrop.js: loadTasks() twice in initTaskSortables()
  * column-element.js: loadTasks(), loadColumns() in multiple functions
  * swimlane-renderer.js: getVisibleTasksForLane() filters tasks multiple times

FRICTION POINT #7: SWIMLANE + STANDARD LAYOUT ARE BRANCHED
============================================================
- renderBoard() checks settings.swimLanesEnabled and calls either:
  * renderSwimlaneBoard (swimlane-renderer.js, 283 lines) OR
  * renderStandardBoard (render.js, 33 lines, local function)
- But swimlane-renderer.js creates task elements via createTaskElement() from task-card.js
  AND swimlane logic is scattered:
  * swimlanes.js: grouping & state
  * swimlane-renderer.js: DOM construction
  * column-element.js: collapse buttons (for both standard AND swimlane modes)
- No clean abstraction: callers must import from 3 modules to render swimlanes

FRICTION POINT #8: ACTIVITY LOG IS A PURE FACTORY (shallow)
============================================================
- activity-log.js: 21 lines, 2 exports (createActivityEvent, appendTaskActivity)
- No state, no side effects — just validation + object construction
- Caller must import both activity-log.js AND use appendBoardEvent() from storage.js
- Example in tasks.js line 56:
  return appendTaskActivity(task, createActivityEvent(type, details, DEFAULT_HUMAN_ACTOR, at));
- Not a friction by itself, but contributes to chattiness

FRICTION POINT #9: NORMALIZE.JS IS A UTILITY ISLAND
===================================================
- 127 lines, 8 exports (validators + coercion functions)
- No state, pure functions
- Storage.js calls normalizers on import (normalizePriority, normalizeSubTasks, etc.)
- But modules can also import normalizers directly (e.g., task-modal.js)
- Duplicate normalization logic in some places vs. lazy normalization in others

SUMMARY OF DEEPENING OPPORTUNITIES
===================================

1. CONSOLIDATE STATE MANAGEMENT
   - Extract storage.js's state-shape + CRUD API into a "StateStore" interface
   - Separate IDB persistence layer from in-memory state
   - Create a "BoardContext" that loads all per-board data in one call
   - Eliminate the 16 load*/save* functions; replace with getState(boardId) / setState(boardId, state)

2. EXTRACT MODAL STATE INTO A SEPARATE MODULE
   - Move selectedTaskLabels, returnToTaskModalAfterLabelsManager into modals-state.js
   - Let task-modal.js import and mutate that state, not own it
   - Reduces task-modal.js from 790 → ~600 lines
   - Makes state testable without rendering

3. CENTRALIZE SWIMLANE LOGIC
   - Merge swimlanes.js + swimlane-renderer.js + swimlane-specific code in column-element.js
   - Create a single SwimLaneRenderer class that encapsulates grouping + DOM construction
   - Rename swimlanes.js to swimlane-state.js (just state + config)
   - Reduce duplication and cross-module coupling

4. BATCH STATE LOADS
   - Create BoardDataLoader() that loads all 4 entities (columns, tasks, labels, settings) in one call
   - Cache the result so multiple modules can share it without re-loading
   - Reduce chatter from 16 modules each doing their own load* calls

5. EXTRACT FILTER LOGIC FROM render.js
   - taskMatchesFilter() should be in a search or filter module
   - Keep render.js as pure orchestration

6. LAZY-LOAD RENDER MODULES
   - render.js, swimlane-renderer.js, column-element.js, task-card.js are only used by render.js
   - Consider merging them or creating a "render-orchestrator" that is lighter
   - Or keep them but move render.js entry point into a thinner kanban.js caller

7. SIMPLIFY MODALS FACADE
   - modals.js does NOT do much (118 lines for mostly re-exports)
   - Either flatten it into kanban.js (Escape handler + cross-modal state) or
   - Keep it but move setupModalCloseHandlers + cross-modal state into a separate "modal-coordinator.js"

8. DOCUMENT TASK SHAPE
   - Tasks have: id, title, description, column, dueDate, priority, labels, relationships, 
     subTasks, activityLog, creationDate, order, text (legacy), collapsed (for swimlanes)
   - No single schema module; validation scattered across normalize.js + tasks.js
   - Create a tasks-schema.js with validation + default object

1. storage.js: god module (1,161 lines, 39 exports)
Six concerns in one file — IDB lifecycle, in-memory state, board CRUD, entity CRUD, cross-board reads, and a 400-line ID-remapping function. Every module imports from it.

2. Scattered render trigger (15+ emit(DATA_CHANGED) sites)
Any mutation anywhere fires a zero-payload signal that causes a full board re-render. dragdrop.js already bypasses it with a dynamic import for partial updates — two patterns, same job.

3. Board normalization split across three modules
schema.js creates objects, normalize.js coerces fields, storage.js remaps IDs — three places own the Task shape. They don't use each other consistently.

4. swimlane-renderer.js leaks swimlanes.js internals
swimlane-renderer.js imports internal accessors from swimlanes.js. render.js branches on swimLanesEnabled with two parallel render paths that share no interface.

5. task-modal.js: DOM-state hybrid (790 lines)
Modal rendering, form state, and cross-modal coordination all live together. modals.js reaches in via leaked getters (getSelectedTaskLabels etc.) with no encapsulation.
